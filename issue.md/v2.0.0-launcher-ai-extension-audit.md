## Findings

### 1. Launcher AI 首次可用性是死路，缺少前置条件引导

落点：

- `src/renderer/src/launcher-shell/LauncherApp.tsx:137`
- `src/renderer/src/ai-core/useAiThread.ts:98`
- `src/renderer/src/ai-core/LauncherAiConversation.tsx:8`
- 对照：`src/renderer/src/components/chat/ChatContainer.tsx:153`

现象：

- 真窗手测里，直接从 launcher 进入 AI 并发送内容，页面只显示原始错误：
  `Model provider credentials are not configured: dashscope`
- 当前 launcher AI 空态只会把错误文本打出来，没有“去设置 provider / 选择 workspace”的下一步入口。
- 主窗口聊天页已经有 workspace 空态 CTA，但 launcher AI 没有把这套前置条件收口成可操作的 setup flow。

为什么重要：

- 这是 AI 基础能力的首屏体验问题，不是锦上添花。
- 对新用户来说，AI 入口“能打开但不能走通”，而且报错是底层 provider 术语，不是产品语言。

最小修正方向：

- launcher AI 不一定要在进入前做硬拦截，但绝不能把用户带进死路。
- 允许用户进入 AI 页面；一旦遇到未配置 model / 未选择 workspace，必须给出明确 setup 引导和 CTA：
  - 未配置模型时，给出“打开 Provider 设置”
  - 未选择 workspace 时，给出“选择 workspace”
- 这层应在 launcher AI host 边界收口，不要把原始 provider 错误直接暴露给用户。

### 2. `Tab` 的“快捷问 AI”语义和当前 BDD 契约不一致，文档/测试表达需要收口

落点：

- 失败用例：`tests/bdd/features/app-launch.feature:18`
- 断言：`tests/bdd/steps/app-launch.steps.ts:218`
- 触发逻辑：`src/renderer/src/launcher-shell/hooks/useLauncherSearchPage.ts:212`
- 清空输入：`src/renderer/src/lib/ai-invocation.ts:92`

现象：

- 本轮 BDD 唯一失败项就是这个场景。
- 从 Home 输入 `整理本周计划` 后按 `Tab`，会走 `initialAction: "submit"`，提交后立刻 `setDraftInput("")`。
- 结合当前产品意图，这其实是合理的：
  - 回车/点击：进入 AI 页面继续编辑
  - `Tab`：快捷问 AI，直接发出
- 现在的问题不是实现和预期不一致，而是仓库内仍有一个 BDD 场景把 `Tab` 写成“进入 AI 后保留 query”。

为什么重要：

- 这会让测试结论误导产品判断，把一个刻意设计的快捷动作误判成回归。
- 也说明 launcher 里“Enter 是进入，Tab 是快捷发送”的交互语义还没有被测试和文案明确表达出来。

最小修正方向：

- 把 `Tab` 明确定义为“快捷问 AI”，不要再把它写成“进入 AI 并保留输入”。
- 更新对应 BDD 场景和命名，让测试验证“消息已提交 / AI 已开始响应”这类可观察结果，而不是验证输入框保留 query。
- 如果后续仍有体感问题，应该查的是“快捷发送后的确认反馈够不够清晰”，而不是 draft 是否被清空。

### 3. Extension 的 navigation capability 契约没有 fail fast，基础边界会被静默绕过

落点：

- `src/renderer/src/extension-host/sdk.ts:194`
- `src/renderer/src/extension-host/PassiveCommandHosts.tsx:53`

现象：

- 官方 SDK 的 `useNativeExtensionNavigation()` 没有沿用 host 层的 capability 校验，而是把缺失能力降级成 no-op：
  - `goHome: () => {}`
  - `hideLauncher: () => Promise.resolve()`
  - `openCommand: () => {}`
- `PassiveCommandHost` 又无条件把 `navigation` 注入 host value，即使 manifest capability 不包含 `navigation`。

为什么重要：

- 这是 extension 基础能力边界的问题。
- 现在的失败模式不是“声明错了立刻炸”，而是“扩展看起来能调用，运行时悄悄什么都不做”。
- 这种静默失败会让 extension author 很难定位问题，也会让 capability 声明失去约束力。

最小修正方向：

- SDK 层恢复 capability 约束，优先复用 `NativeExtensionHost.tsx` 里已经存在的 fail-fast 逻辑。
- `PassiveCommandHost` 只在 manifest 明确声明 `navigation` 时注入 navigation。
- `push/pop` 仍可作为 view stack 能力单独扩展，但不要顺手把 launcher navigation 一起放宽。

### 4. Launcher 搜索结果会驱动窗口高度连续跳变，真实窗口已测到 `68 -> 424 -> 562`

当前状态：

- #8 已先补出稳定 search shell、section、loading 和 stale preview 不可执行。
- 已补：非空 query 统一使用固定搜索结果 viewport，`resultsViewportHeight` 不再跟随 provider 分批结果数量重算。
- 这条 v2 的验收标准不是“Raycast 逐像素一致”，而是“搜索态固定高度，结果在固定 viewport 内填充/滚动”。

落点：

- 搜索 fan-out：`src/renderer/src/launcher-shell/hooks/useLauncherSearchPage.ts:149`
- 高度计算：`src/renderer/src/launcher-shell/hooks/useLauncherSearchPage.ts:110`
- 窗口 resize：`src/main/windows/launcher-window.ts:231`

现象：

- 真实 Electron 手测里，输入 `todo` 后，launcher 窗口高度在一个查询过程中出现了三个离散值：
  - `68`
  - `424`
  - `562`
- 根因不是单个动画参数，而是 renderer 对每个 source 分开发 IPC 搜索，结果分批回流；每批结果都会重新算 `viewportHeight` 并立刻发到 main process 调整窗口。

为什么重要：

- 这是明显的体感问题，用户已经能感觉到“搜索时高度不一致抖动”。
- 它还会放大结果列表、footer、surface 切换之间的视觉不稳定。

最小修正方向：

- 不要让窗口高度跟着每个 provider 的分批结果实时变化。
- 两个可行最小修法：
  - 搜索态锁定一个稳定高度，等结果稳定后再调整
  - 把多 source 查询在主进程合并后一次回传，避免 renderer 级 fan-out
- 如果仍保留渐进式结果，至少要把窗口高度和结果渲染解耦。

### 5. Launcher 不能通过自己的命令系统打开 Settings，键盘主路径断了

当前状态：

- 产品复判：Settings 有稳定入口和快捷键即可，这条不再作为 v2 阻塞。
- 当前主干已经提供 Settings 入口与 `searchOpenSettings` 快捷键。
- 保留明确可见的入口即可；搜索结果里必须出现 Settings command 不是 v2 必须项。

落点：

- `src/renderer/src/launcher-shell/pages/index.ts:15`
- `src/shared/shortcuts/ids.ts:1`
- `src/renderer/src/launcher-components/LauncherSearchPage.tsx:146`

现象：

- `listLauncherCommands()` 只索引 built-in AI 和 extension commands，没有 settings command。
- 默认快捷键里也没有 “open settings” 这类 launcher 内命令。
- 真窗手测里，搜索 `settings` / `设置`，结果都是：
  - `问 AI`
  - 浏览器历史
  - 文件
    但没有应用自己的设置入口。
- 当前唯一稳定入口是 footer 里的鼠标按钮。

为什么重要：

- 这是产品边界问题，不只是“少一个功能”。
- 如果 launcher 是主入口，它至少应该能打开和解释它自己的设置。
- 现在的体验更像“一个能搜别人的东西的壳”，而不是“能自洽控制自己的产品入口”。

最小修正方向：

- 把 Settings 收进 launcher command 体系，允许搜索和快捷键直接打开。
- 不要只把设置入口藏在 footer 按钮里。
- 建议把 settings 视作 first-party command，而不是窗口级特例。

### 6. 首页 history grid 是网格视觉，但键盘语义仍然是线性列表

落点：

- `src/renderer/src/launcher-components/LauncherHistoryGrid.tsx:23`
- `src/renderer/src/launcher-shell/hooks/useLauncherSearchPage.ts:218`

现象：

  - `ArrowUp`
  - `Enter`

- history grid 需要独立的二维选择模型：
  - `Left/Right` 按列移动
  - `Up/Down` 按行移动
- 不要继续复用 result-list 的 `delta ±1` 选择逻辑。

落点：

- `src/renderer/src/ai-core/LauncherAiPage.tsx:174`
- `src/renderer/src/ai-core/LauncherAiPage.tsx:224`
- `src/renderer/src/ai-core/LauncherAiPage.tsx:254`

  - footer 动作按钮：`h-7`，28px

### 9. 工具执行分组的展开态不归用户所有，新的 event / approval 会接管并重置它

落点：

- `src/renderer/src/components/chat/message-projection.ts:110`
- `src/renderer/src/components/chat/Messages.tsx:281`
- `src/renderer/src/components/chat/Messages.tsx:292`
- `src/renderer/src/components/chat/Messages.tsx:317`
- `src/renderer/src/components/chat/Messages.tsx:355`
- `src/renderer/src/components/chat/ChainOfThought.tsx:53`

现象：

- tool cluster 的 key 是 `tools:${assistantMessageIds.join(":")}`。同一轮执行里只要又流入一个带 `tool_calls` 的 assistant message，这个 key 就会变化，整段工具分组 remount。
- grouped tool item 的 key 还把审批态编码进去了：`pending/done`。审批一旦从待处理变成已处理，这一行也会 remount。
- 分组本身的 open state 不是用户主导，而是：
  - 默认跟着 `hasActiveActions` 自动展开
  - `collapseWhenInactive` 在执行结束时自动收起
  - remount 后 `openOverride` 直接丢失
- 结果就是：用户刚手动展开或收起，后面来了一个新 event、一个 tool result，或者审批状态一变化，UI 就可能重新展开、收起，或者丢掉当前 inspection 上下文。

为什么重要：

- 这不是“动画参数不舒服”，而是控制权错位。
- 对工具流来说，展开态本质上是用户的 inspection 决策；现在它被运行时事件接管了，所以体感上就会像“抖”“自己在动”“不受控”。
- 这类问题会直接削弱 AI/agent 的可信度，因为用户会怀疑自己看的到底是不是同一段执行。

最小修正方向：

- 先定义展开态归属：它应该挂在稳定的 tool run / tool call 身份上，而不是挂在瞬时列表结构上。
- tool cluster 和 grouped item 都应改成稳定 key，不要把 `pending/done` 这类瞬时状态编码进 key。
- 只有首次出现时允许用运行态决定默认开合；一旦用户手动操作过，后续 event 不应再擅自改写。

### 10. 同一段工具执行会在进行中从“单条消息”突然变成“折叠分组”，结构身份不稳定

落点：

- `src/renderer/src/components/chat/Messages.tsx:313`
- `src/renderer/src/components/chat/Messages.tsx:340`
- `src/renderer/src/components/chat/Messages.tsx:355`
- `src/renderer/src/components/chat/Messages.tsx:406`

现象：

- 当前逻辑是：
  - 只有 1 个 tool call 时，直接渲染单条 `ActionMessage`
  - tool call 数量达到 2 个时，整段切换成 `ChainOfThought` 折叠分组
- 但 agent/tool 执行是流式到达的，所以同一段执行完全可能先显示成一条普通工具消息，过一会儿第二个 tool call 到了，又突然长成一个可折叠容器。
- 从用户眼里看，这不是“内容变多了”，而是同一个对象中途换壳，点击区域、标题、摘要和空间占位都会跟着变。

为什么重要：

- 这是结构级抖动，比单纯的高度跳变更破坏控制感。
- 如果主产品想把工具流做成“可检查、可理解的执行时间线”，那它就不该在执行过程中频繁改 UI 语法。

最小修正方向：

- 同一段工具执行从第一帧起就该有稳定容器语义。
- 两个可行方向：
  - active tool run 从第一条 tool call 开始就统一占用 grouped shell
  - 或者始终使用一致的单条 timeline 容器，只在内部逐步长出更多步骤
- 不要让“tool call 数量跨过 1 -> 2”成为 UI 结构切换阈值。

  - `command-intents`
  - `search-results`
  - `suggestions`
- 但当前只有 `suggestions` 会显示 section header，其余几类都是裸项直接混排。
- 用户只能靠每一行左侧很小的 categoryLabel 去猜这条是命令、意图、还是普通搜索结果。

为什么重要：

- Raycast 官方 `List` 文档明确把 “用 section 的 title/subtitle 分组相关项” 当成默认 UX 能力。
- 现在 launcher 结果类型已经异构化了，但界面还是按单一列表在画，扫描成本高，尤其在 query 同时命中命令、文件、历史、建议时会显得很乱。
- 这也会削弱前面提的“稳定 viewport + section 填充模型”，因为 section 现在只有“数据结构上有”，用户视觉上没有。

最小修正方向：

- 至少把 `commands / intents / search results / suggestions` 这几类 section 以明确标题显示出来。
- section 标题不需要很重，但必须让用户知道当前列表是在“哪类结果之间切换”，而不是只在行级读小标签。
- 如果后续要继续扩来源，先守住 section 语义，不要再往无标题混排里堆数据源。

### 13. Home 异步搜索没有显式 loading 状态，用户只能靠列表跳变和窗高跳变感知系统在工作

落点：

- `src/renderer/src/launcher-components/LauncherChrome.tsx:17`
- `src/renderer/src/launcher-components/LauncherInput.tsx:109`
- `src/renderer/src/launcher-shell/hooks/useLauncherSearchPage.ts:59`
- `src/renderer/src/launcher-shell/hooks/useLauncherSearchPage.ts:155`
- `src/renderer/src/launcher-shell/hooks/launcher-search-page-store-core.ts:17`

现象：

- launcher chrome / input 本身已经支持 `inputStatus`，并且有 pending/tooling 指示器。
- 但 Home 搜索页没有把 search request 状态接到 `LauncherChrome`，store 里也没有 pending / settled 这类显式状态。
- 结果就是：异步搜索开始后，用户看不到“正在搜”，只能通过列表内容变化和外层高度变化去猜系统是否还在回流结果。

为什么重要：

- Raycast 官方 `List` 把 `isLoading` 作为一等能力，明确用在搜索栏下方的 loading bar，而不是让列表自己抖来表达异步。
- 现在 Openwork 的体感问题被放大了一层：
  - 高度在抖
  - 列表在变
  - 但系统没有明确说“我还在搜”
- 这会让人把渐进式回流误解成不稳定，而不是受控的异步搜索。

最小修正方向：

- Home 搜索页需要一个明确的 search request 状态，而不是只有 `resultsBySource`。
- 搜索进行中时，把状态接到现有 `inputStatus` / header cue，上屏一个稳定 loading affordance。
- 这条应该和高度稳定一起做：`稳定 viewport + 明确 loading`，而不是只做其中一半。

### 14. Root Search 的命令发现能力比 Raycast 基准弱一截，没有真正的 fuzzy + alias 语义

落点：

- `src/renderer/src/launcher-shell/search-items.ts:97`
- `src/renderer/src/launcher-shell/search-items.ts:121`
- `src/renderer/src/launcher-shell/hooks/launcher-search-page-store-core.ts:85`
- `src/renderer/src/launcher-shell/types.ts:9`

现象：

- Raycast 官方 `Search Bar` 文档明确写的是 root search fuzzy search，可匹配 title / subtitle / keywords / alias。
- 当前 Openwork 的 first-party command 搜索主要还是：
  - title prefix
  - title substring
  - keyword exact / includes
  - ownerTitle / description includes
- cached search results 的本地过滤更窄，只看 `title + subtitle + id` 的普通 includes。
- 目前也没有用户可见的 alias 语义，launcher item 数据结构里也没有这层归属。

为什么重要：

- 这不是“再多加几个关键词”的问题，而是 root search 的核心手感。
- 如果 launcher 要成为默认入口，用户需要能用很短、很模糊、很个人化的记忆碎片把命令找出来。
- 现在这套匹配方式对固定句法还行，但离 root search 级别的“模糊可达”还差一截。

最小修正方向：

- 先把 “root search 对哪些字段负责匹配” 定义清楚：
  - title
  - subtitle
  - keywords
  - alias
- 在 first-party command / internal command 上优先补一套真正的 fuzzy 匹配，再决定是否暴露用户别名配置。
- 不要继续把 root search 质量寄托在 scattered keywords 和 substring 命中上。

## 产品判断

这轮已经定下来的产品判断是：

- `Launcher` 是 Openwork 的默认控制面，不是附属搜索框
- `Home` 在 query 非空后进入稳定的 `search shell`
- `Home` 结果页必须具备 `action / section / loading` 三件套
- `Launcher AI` 允许进入，但不能把用户带进死路；未满足前置条件时必须给 setup 引导
- `工具时间线` 的展开态默认可由系统驱动，但用户一旦手动展开/收起，后续控制权就归用户

现在最突出的矛盾是：

- Launcher 想当主入口
- 首页是网格，键盘却还是列表
- Translate 自然语言入口在中文里不稳
- Launcher AI setup gate 还没有把裸 provider/workspace 错误转成明确 CTA
- Home 搜索页还没有接出完整 action panel
- root search fuzzy / alias 还没有达到 Raycast 式 root search 手感

已收口的部分不要再反复摇摆：

- Settings 放 footer 左下角是合适的，不要求进入搜索结果列表。
- Home 非空 query 进入固定高度 search shell，结果在固定 viewport 内填充/滚动。
- 工具时间线展开态和容器身份已经通过 PR #9 收口，后续只在真实复测发现问题时再开新项。

剩余项不是“以后再增强”的泛化问题，而是上述几个产品定义还没跟上。

## 对标与实施约束

Raycast 和 macOS Spotlight 给出的共同信号不是“严格固定高度多少 px”，而是：

`搜索态是稳定 viewport，结果只是往里面填。`

对 launcher 搜索壳体，建议按这个边界实施：

- `Home` 可以保留紧凑高度
- 只要 query 非空，就切进稳定的 `search mode`
- `search mode` 高度跟页面语义走，不跟 provider / extension 的分批结果回流走
- 结果少时，不缩窗，靠 section 填充信息密度：
  - 精确结果
  - `问 AI`
  - `Search with xxx`
  - extension commands
  - history / files / recent / suggestions
- 结果多时，在稳定 viewport 内滚动，而不是持续改外层窗口高度
- 真正切到 `AI/detail/page` 这类不同页面语义时，才允许切换到另一档高度

样式方向：

- 视觉节奏参考 Raycast：稳定壳体、清晰 section、弱装饰、强可扫读性
- 不追求照搬 Raycast 皮肤，但应避免当前这种因搜索回流造成的结构和高度抖动

当前实现约束也需要一起记住：

- 通用 icon 库统一使用 `lucide-react`，不要再引入第二套通用 icon 库
- 如果后面要补少量更合适的 icon，可以用已安装好的 `better-icons` 做检索，但最终仍应优先落在 `lucide-react` 或局部 SVG
- 前端运行时依赖增删后，按 `AGENTS.md` 要跑 `npm run audit:frontend-packages`
- 打包链路或运行时依赖调整后，至少验证 `npm run typecheck`；影响产物装配时再补 `npm run build`
- `electron.vite.config.ts` 已显式开启 `main / preload / renderer` 的 `rollupOptions.treeshake = true`，后续不要引入破坏 tree-shaking 的重副作用依赖

## 建议补的测试

建议新增 BDD，优先覆盖用户感知最强的行为边界：

- Launcher 搜索过程中窗口高度不应出现多次跳变
- Home history grid 应支持二维方向键导航
- Launcher footer 左下角 Settings 入口应稳定可见并可打开 Settings
- Translate 中文自然语言入口可触发翻译页
- Launcher AI 在未配置 model / workspace 时应展示 setup CTA，而不是裸错误
- Home 选中结果时 `⌘K / Ctrl+K` 可打开 launcher action panel，并执行 secondary action
- 搜索 query 同时命中命令、结果、建议时，section 标题应稳定可见
- 异步搜索进行中时，launcher 输入区应展示明确 loading 状态
- first-party command 在 root search 中支持 fuzzy 命中关键词/简写

工具时间线相关项已由 PR #9 补了 node 级结构稳定性测试；下面三条不再作为新实现待办，只作为真实复测或后续回归场景：

- 用户手动收起的 tool group 在后续 tool event / result / approval 变化后仍保持收起
- tool approval 从 pending 变成 resolved 后，当前已展开的工具详情不应丢失
- 同一段工具执行在新增 tool call 时，不应从单条消息突然换成另一套容器语义
