# The Launcher Is The Doorway; The Workflow Is The Product

[中文](./launcher-to-agent-workflow-cn.md)

Launchers are excellent at the first move.

They compress the distance between intention and action. Press a shortcut, type
a few words, open something, search something, start something. That speed is
real product value. A desktop agent should not make people walk through a maze
before work can begin.

But agent work does not end at invocation.

Once an agent starts reading context, planning, using tools, asking for approval,
and producing changes, the product question shifts. It is no longer only "how
fast can I start?" It becomes:

- what context is the agent using;
- what is it doing now;
- what is risky;
- what changed;
- what needs my decision;
- what evidence did it leave;
- how do I recover or continue later?

That is why the launcher should be treated as the doorway, not the whole house.

The launcher wins the beginning. The workflow earns the trust.

This distinction matters because it prevents a common product trap: wrapping a
black-box agent in a beautiful command palette and calling it done. That gives
the user a fast start, but it abandons them during the actual work.

Delegated work needs a surface after the first prompt. It needs progress that
can be inspected, decisions that can be approved or rejected, outputs that can be
reviewed, and history that can be resumed.

The design principle is simple:

keep the first move fast, and keep the rest inspectable.

That is the difference between a launcher with AI and an agent workspace.
