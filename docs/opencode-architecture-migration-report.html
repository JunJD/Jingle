一页结论 优势、差距、迁移方向 opencode 架构 server、sync、plugin、desktop Openwork 当前态 main agent、renderer、extension runtime 迁移路线 从 facade 到 event log 再到 sidecar

一页结论

opencode 优秀在哪儿： 它的核心架构把“agent 能力”定义成可被多个前端消费的服务协议，而不是 Electron 某个窗口背后的实现细节。这个选择直接带来四个结果：边界清楚、状态可重放、插件 hook 能扎在核心执行路径上、桌面启动和崩溃恢复可以按 sidecar 生命周期治理。
维度 opencode 当前做法 Openwork 当前做法 判断
核心边界 agent/server 是一等核心，桌面通过 sidecar 启动本地 HTTP 服务。 agent runtime 主要在 Electron main 内，由 IPC controller 暴露给 renderer。 not true yet Openwork 还没有稳定的 agent core protocol。
客户端契约 HttpApi/OpenAPI/SDK/SSE 是产品内公共契约。 preload IPC + renderer thread store 是主要契约。 partially true 有类型 IPC，但不够跨客户端。
状态模型 SyncEvent：typed event、aggregate、version、seq、projector、replay。 Prisma Thread/Run/HITL/Artifact + LangGraph checkpoint/write。 partially true Openwork 有持久化，但不是 event-first。
插件位置 插件 hook 在 config、auth、provider、chat、permission、tool、shell 等核心路径。 extension runtime 偏 UI/host capability；agent tool 侧有 native extension AI capability。 partially true UI extension 强，core hook 弱。
桌面生命周期 Electron main 管 sidecar port、auth、health、migration、window 创建。 Electron main 同时管窗口、服务注册、agent runtime、projection、extension runtime。 not true yet Openwork agent core 未从桌面壳拆出。
GUI harness opencode desktop/web/tui 是 core 的客户端，GUI harness 不是主要差异化。 主产品就是 launcher-first + harness-first 的 delegated work：审批、artifact、记忆、恢复。 already true Openwork 产品层不是落后项。

最值得迁移的东西

  1. 先把 Openwork agent core 收口成一个内部 API facade，不立刻 sidecar 化。
  2. 引入 typed event log + projector，先投影现有 AgentStreamHub，不要替换 checkpoint。
  3. 做 renderer SDK adapter：当前 IPC 是一个 transport，未来 HTTP/SSE 是另一个 transport。
  4. 把 model provider 拆成 catalog / auth / policy / secrets / user settings 的明确边界。
  5. 把 extension capability 与 core plugin hook 分开：前者管 UI/host，后者管 agent 执行路径。

不该照搬的东西

  1. 不要为了学 opencode 先迁移到 Bun 或 Effect；收益来自边界，不来自库本身。
  2. 不要马上替换 DeepAgents/LangGraph loop；Openwork 的 HITL、artifact、memory 都依赖当前 checkpoint 语义。
  3. 不要把 extension UI runtime 改造成 opencode plugin；二者职责不同。
  4. 不要先做 sidecar。没有 protocol 和 parity test，sidecar 只会把调试半径扩大。

opencode 架构拆解

opencode 是 monorepo：packages/opencode 承载 core agent/server/cli， packages/app 是 Solid web client，packages/desktop 是 Electron shell， packages/sdk/js 和 packages/plugin 是外部契约。这个布局本身已经说明： core 不是 UI 的附属物。

Desktop Shell 选择端口、生成密码、启动 sidecar、健康检查、再创建窗口。
Local Server HttpApiApp + Effect layer，暴露 HTTP、SSE、WebSocket、OpenAPI。
Domain Core Session、Provider、Permission、MCP、PTY、Workspace、Sync。
Event / Sync typed events、aggregate seq、projector、replay、bus publish。
Clients Web App、TUI、desktop renderer、SDK 都通过协议消费 core。

1. 本地服务是一等边界

packages/opencode/src/server/server.ts 定义了可直接调用的 in-process Default app，也定义了 listen(opts) 形式的监听器；同一个 core 既能被测试/内联调用， 也能作为本地服务运行。关键证据在 server.ts:58-83 和 server.ts:103-140：HTTP routes、WebSocket tracker、server layer、ConfigProvider 都是 listener layer 的组成部分。

Desktop 没有把 agent loop 写进窗口生命周期里。packages/desktop/src/main/index.ts:287-354 先找本地端口、生成密码、spawn sidecar、等待 health，再在 index.ts:369-374 创建主窗口。 sidecar 的启动细节在 packages/desktop/src/main/server.ts:69-180：utility process、startup timeout、ready/error/exit、health wait 都是清晰的生命周期事件。

2. API 是产品内公共契约，不是 renderer 私有通道

packages/opencode/src/server/routes/instance/httpapi/api.ts:26-59 把 BusEvent/SyncEvent schema 汇入 HTTP API，并把 Config、File、MCP、Permission、Provider、Session、Sync、Workspace、V2 等分组挂到同一个 OpenCodeHttpApi。这个设计意味着 UI 不直接 import core internals，而是通过 API shape 感知系统能力。

前端通过 SDK 和 SSE 连接服务。packages/app/src/context/server-sdk.tsx:30-34 创建 SDK；server-sdk.tsx:119-200 管 reconnecting SSE loop、heartbeat、coalescing、delta 去重； server-sdk.tsx:223-244 返回 client/event/createClient。这个 client 层是 Openwork 现在最缺的中间层： 它把 transport、重连、批处理和 UI 状态消费拆开。

3. SyncEvent 给状态变化一个可重放的骨架

packages/opencode/src/sync/index.ts:24-66 定义了 event definition：type、version、aggregate、schema、properties， 以及 run/replay/replayAll/remove/claim 接口。sync/index.ts:75-115 在 replay 时检查 latest seq、owner、expected seq； sync/index.ts:136-170 在 immediate transaction 里生成 ascending id 和 seq，再进入 projector。

这不是“事件总线换个名字”。它同时承担三个职责：写入顺序、投影一致性、跨进程/跨客户端同步。Openwork 当前有 checkpoint 和 DB，但还缺一个“用户可见状态变化”的一等事件层。

4. 插件 hook 扎在 core policy 和执行路径上

packages/plugin/src/index.ts:56-74 规定 plugin 输入包含 client、project、directory、worktree、serverUrl、shell。 index.ts:222-334 列出 hooks：event、config、tool、auth、provider、chat.message、chat.params、permission.ask、 command.execute.before、tool.execute.before/after、shell.env、tool.definition 等。

这组 hook 的价值在于它们不只是“扩展能画 UI”，而是能在 provider、permission、tool execution、chat params 等核心节点影响 agent 行为。Openwork 的 extension runtime 已经有 host capability 隔离，但 core hook 层还没有同等清晰的公共契约。

5. Provider catalog 与 policy 开始拆开

opencode v2 的 packages/core/src/catalog.ts:43-83 把 provider/model/default model 作为 catalog state；catalog.ts:191-198 在 finalize 阶段触发 catalog.transform 并应用 provider.use policy。specs/v2/provider-policy.md:14-20 明确 provider config 和 provider policy 是不同关注点；provider-policy.md:77-99 说明 last matching statement wins；provider-policy.md:192-202 又明确 plugins 不能覆盖 policy。

这对 Openwork 的模型服务很有启发：provider endpoint、auth secret、model catalog、默认模型、组织/用户 policy 不应该挤在一个“设置服务”语义里。

Openwork 当前态

这里按当前 checkout 说，不沿用旧印象：当前 PromptInput 已经有 composer mode，ComposerArea 使用 Lexical 和 source mention；bundled extension registry 当前静态接入 Apple Reminders、GitHub、Notion、todo-list 与 translate。 Apple Reminders / GitHub / Notion 已经按 package entry 进入 extensions/，todo-list / translate 仍在 src/extensions 下。

Openwork 已经强的部分

  • GUI harness 是主产品能力：线程、审批、artifact、memory、fork/recovery 都围绕 delegated work 生命周期设计。
  • LangGraph checkpoint 让运行恢复、HITL interrupt、thread projection 可以工作，不能粗暴替换。
  • 个人记忆是本地 SQLite/Prisma product data，不是远端优先服务；这个产品语义很关键。
  • extension runtime 已经有 utility process / session / host capability 边界，比普通插件 UI 更像桌面平台。
  • renderer 侧 thread store/projection 已经形成一套产品状态模型，迁移时要保护 UI 行为。

Openwork 当前短板

  • agent core 主要嵌在 Electron main，跨客户端/跨进程协议不是第一层设计对象。
  • IPC controller 暴露的是 Openwork renderer 需要的投影通道，不是稳定 SDK/API。
  • 状态持久化是 Thread/Run/HITL/Artifact/checkpoint 混合模型，还没有 typed event log 作为 replay/source of truth。
  • extension runtime 偏 host UI capability，缺 core plugin hook 的统一扩展点。
  • provider/model/policy 的边界可继续拆清楚，特别是组织/工作区/用户优先级和禁止使用策略。

1. Main composition root 是桌面平台入口，但 agent core 也在里面

src/main/composition-root.ts:95-120 注册 agent、artifact、extension runtime、launcher、model provider、memory、threads 等 IPC handlers；composition-root.ts:122-134 启动 native menu bar、extension runtime menu bar、native island status、shortcut、launcher search。 这是清楚的 Electron main composition root，但也说明桌面进程承担了太多核心服务编排。

2. AgentService 是事实上的 agent core

src/main/agent/service.ts:419-520 展示了当前 invoke 主路径：activeRuns、workspace 解析、permission mode、extension AI capability、 memory context、begin run。src/main/agent/runtime.ts:150-216 创建 LangGraph/DeepAgents runtime，装配 model、checkpointer、 LocalSandbox、extension AI runtime；runtime.ts:270-315 把 todo、filesystem、artifact、web、summarization、patch、serialized tool call、 skills、desktop automation、extension AI、guardrail、tool approval 串成 middleware stack。

这是 Openwork 的产品核心，不应该被“换成 opencode processor”一刀切掉。更合理的方式是先在它外面包一层 Openwork Agent Core API， 然后逐步把事件和 transport 从实现里拉出来。

3. 当前 transport 是 IPC projection

src/main/agent/controller.ts:22-56 注册 agent:invoke、agent:resume、agent:cancel、 agent:getProjection、agent:subscribeProjection。controller.ts:98-120 把 projection envelope 通过 agent:projection:${threadId} 发回 renderer。

renderer 侧 src/renderer/src/lib/thread-context.tsx:136-170 把 projection 写回 thread store、streamData、loading state， 并在 loading 结束或 approval 变化时刷新 fork state。这个模型是可用的，但它把 UI 状态消费和 main-side projection 绑定得比较紧。

4. 持久化是 product data + checkpoint，不是 event source

prisma/schema.prisma:10-30 定义 Thread，schema.prisma:32-51 定义 Run， schema.prisma:140-162 定义 HitlRequest，schema.prisma:164-193 定义 Artifact， schema.prisma:210-240 定义 Checkpoint/CheckpointWrite。src/main/agent/persistence.ts:39-77 begin run 时写 Run 和 Thread busy；persistence.ts:192-231 从最新 checkpoint 同步 run/thread status 和 title。

这套模型适合 Openwork 当前产品：checkpoint 负责 runtime 恢复，Prisma 表负责产品查询。但它不像 opencode SyncEvent 那样天然回答： 某条消息、某个 tool call、某个审批、某个 artifact 是由哪个事件、按什么顺序变成现在这样的。

5. Extension runtime 是 Openwork 自己的优势，不应和 opencode plugin 混成一层

src/main/services/extension-runtime/runtime-manager.ts:47-84 定义 host capabilities：AI、clipboard、storage、toast、navigation、 native extension invoke、quicklink、preferences、external link 等。runtime-manager.ts:124-143 管 RuntimeSession，包括 kind、process、runtimeCapabilities、sessionId、stopping。

这层负责“扩展 UI/runtime 如何安全地和宿主交互”。opencode plugin hook 负责“agent core 的 provider/tool/permission/chat 如何被扩展”。 迁移时应该把它们并列，而不是互相替换。

真正差距：不是 UI，是 core contract

问题 opencode 给出的答案 Openwork 应该怎么吸收
如果明天要加 Web client、CLI client 或自动化测试 client，怎么连 core？ 已有 HTTP API、SDK、SSE event。 定义 AgentCoreClient，先让 renderer 通过 IPC adapter 使用，再补 HTTP/SSE adapter。
如果 UI projection 错了，如何重建？ SyncEvent replay + projector。 先为 run/message/approval/artifact 引入 append-only event table，投影到现有 AgentStreamHub。
插件想改 provider 或 tool 行为，hook 在哪儿？ provider/auth/chat/tool/permission/shell hooks 是 core API。 不要塞进 extension UI runtime；单独定义 agent-core hook registry。
桌面进程崩了或 agent core 卡了，如何定位？ sidecar process、health endpoint、startup timeout、server logs。 先做 health/status API 和 structured lifecycle，再考虑 utilityProcess sidecar。
provider 可用性是设置、密钥还是策略？ catalog/config/auth/policy 明确分层，policy 不能被 plugin override。 把 Openwork model-provider 继续拆边界：catalog 只描述能力，policy 决定能否使用。

迁移路线

迁移目标不是“把 Openwork 变成 opencode”，而是保留 Openwork 的 GUI harness 和本地记忆语义，把 agent core 从 Electron main 的实现细节提升成可验证、可重放、可多客户端消费的核心协议。

Phase 0：冻结目标边界

边界定义：Openwork target = Electron shell + local agent service + typed event/API contract + renderer SDK adapter。

  • Electron shell 只管窗口、快捷键、系统菜单、更新、sidecar/服务生命周期。
  • Agent core 管 thread/run/session、tool、permission、memory context、artifact event、provider selection。
  • Renderer 不直接理解 main internals，只消费 client + events + projection。

验收：补一页架构边界文档或 ADR，把“哪些状态归 main，哪些归 renderer，哪些是可重放事件”写清楚。

Phase 1：Agent Core Facade，不改 runtime

在 src/main/agent-core 或 src/main/agent/api 定义内部 API： invoke、resume、cancel、getProjection、subscribe、 getThread、listThreads、getHistory。第一版仍调用现有 AgentService、AgentStreamHub、ThreadsService、Prisma checkpoint。

为什么先做这个：它把 transport 和实现拆开，但不碰 LangGraph/HITL/artifact/memory 的行为。

验收：renderer 的 agent IPC handler 只依赖 facade；现有 agent BDD/类型检查通过。

Phase 2：Typed Event Log + Projector，先旁路 checkpoint

增加一张 append-only event table，例如 agent_events： event_id、aggregate_type、aggregate_id、seq、type、 version、payload_json、created_at。先覆盖新事件： run.status.changed、message.part.delta/updated、approval.requested/resolved、artifact.updated、memory.included。

projector 第一版只负责把 event 投影到现有 AgentStreamHub 或一个等价 projection store。 checkpoint 仍是 LangGraph runtime recovery 的 source of truth。不要尝试把历史 checkpoint 全量转事件。

验收：同一个 run 的事件 seq 单调；重建 projection 与现有 thread projection 在关键字段一致。

Phase 3：Renderer SDK Adapter

在 renderer 建一个 OpenworkAgentClient interface：方法调用 + event subscription。第一版实现是 IPC adapter； 第二版可以加 dev-only HTTP/SSE adapter。UI 只依赖 client，不直接依赖 window.api.agent.* 的散点调用。

这一步对齐 opencode 的 server-sdk.tsx 思路：transport 负责重连、心跳、批处理和事件 coalescing； React store 只负责消费 projection。

验收：同一套 renderer store 能在 IPC adapter 下工作；HTTP adapter 在 dev flag 下通过 smoke test。

Phase 4：Model Provider Catalog / Policy 拆层

把 provider 相关职责拆成五块：catalog 描述 provider/model 能力；auth/secrets 管凭证；settings 管用户默认选择； policy 管是否允许使用；runtime resolver 在一次 run 开始时固化 snapshot。

对 Openwork 特别重要的是：policy 不能被 extension 或 workspace 默认配置悄悄覆盖用户全局禁用。这个规则应该先出现在类型和测试里。

验收：新增 provider.use policy evaluator 单元测试；现有设置页不变；run metadata 能记录 provider/model/policy snapshot。

Phase 5：Core Hook Registry

定义 Openwork 自己的 core hooks：tool.execute.before、tool.execute.after、 permission.ask、model.params、message.transform、shell.env、 artifact.created。这些 hook 属于 agent core，不属于 extension runtime UI tree。

extension runtime 可以成为 hook provider 的一个来源，但 hook 执行必须在 main/agent core 里受 policy 和 permission 管。

验收：先让 native extension AI tool 走同一 before/after hook；approval 仍由现有 ToolApprovalMiddleware 兜住。

Phase 6：Sidecar，只在协议稳定后做

当 API facade、event log、renderer client、health status 都稳定后，再把 agent core 移入 Electron utilityProcess sidecar。桌面 main 负责端口、auth、health、crash/restart、window readiness；renderer 通过同一 client 连接，不感知 core 在 main 里还是 sidecar 里。

验收：OPENWORK_HOME 临时目录下的 BDD startup test；sidecar health timeout；异常退出后 UI 能显示明确失败状态；IPC/HTTP parity test。

风险与迁移约束

最大风险：替换 runtime

Openwork 的 LangGraph checkpoint 已经承载恢复、HITL interrupt、memory inclusion、artifact 工具链。 直接替换成 opencode session processor 会破坏产品闭环。

第二风险：先做 sidecar

sidecar 是结果，不是起点。没有 API facade 和 event contract，进程拆分只会让同一个隐式耦合跨进程发生。

第三风险：插件概念混淆

Openwork extension runtime 是桌面 UI/runtime 平台；opencode plugin 是 agent core hook 平台。二者应该桥接，不应该合并。
