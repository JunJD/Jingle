Openwork / Common Ground
1440 桌面视角预览，launcher 常态宽度建议 680-760px
设计不是排方块，而是安排视线和行动。

让空间先建立秩序，组件再补足语义。

这版预览把层级从“卡片堆叠”改成“留白、比例、行距和规则线”。用户先感知的是入口、流向和重心，而不是一堆被描边的模块。对于面向办公群体的 agent 产品，这是更耐看、也更耐用的组织方式。

核心判断
真正需要被框起来的，只有“窗口”本身和“审批”这种高风险节点。其余信息应更多依赖空间关系。
设计重心
Space
留白先定义秩序，卡片后退。
Rhythm
标题、正文、次要信息走清晰节奏，不靠描边堆层级。
Focus
只有当前行动和风险点拿到高对比强调。
Launcher / 意图入口
Launcher 应该像一块被准确放置在桌面上的工作浮层，而不是抽屉式菜单。它不需要很多包裹块，只需要一个清楚的输入重心、几条可靠的结果行，以及足够的呼吸空间。
输入先于分类 用户的眼睛一进来就应落在意图文本上，而不是标签和图标墙上。
结果按线性阅读组织 类型、标题、解释、可执行性在同一行解决，不再人为分成小卡片。
宽度为中文服务 对办公中文输入来说，520 太紧，720 左右更能容纳完整意图和可信描述。

人工确认

是否允许代理把预算变更表附加到外发材料？

这份附件含价格区间和内部假设，一旦外发，会直接影响客户谈判预期。

颜色

只保留一个行动色、一个代理色、一个审批色。其余层次用明度和空间区分，不再让颜色承担过多结构职责。

边界

边界主要出现在窗口、列分隔和审批节点。模块之间更多依靠间距、基线和阅读顺序建立关系。

中文

中文需要更大的横向呼吸空间。launcher 宽度不足时，视觉会迅速碎掉，所以不应拿 command palette 的极限宽度套过来。

字体

中文正文建议用思源黑体子集，路径和命令再用窄一些的等宽字体。字体文件路径可继续沿用 src/renderer/src/assets/fonts/。

这版预览刻意减少“方块式组织”。如果接下来继续往实现走，我会把 design token 收敛成两类约束： 一类控制颜色、字重、间距；一类控制空间规则，例如 launcher 宽度、三栏比例、审批节点的最小强调强度。
