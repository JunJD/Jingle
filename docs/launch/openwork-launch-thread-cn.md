# Openwork 产品预告长推

用途：X / 即刻 / 朋友圈长文预告。  
语气：面向第一次听说 Openwork 的读者，不默认读者懂 agent、harness、extension、HITL。

## 版本 A：完整长推

我们在做 Openwork。

一句话：它是一个让普通人把软件工作交给 AI，但不把控制权交出去的桌面 Agent。

这里的“普通人”不是说能力普通，而是说他不想成为程序员。

创始人、运营、设计师、研究员、产品经理、独立开发者之外的那些人。他们每天都会碰到软件问题：改一个网站、连一个 API、写一个脚本、整理一批数据、修一个自动化流程、把一个内部工具搭起来。

他们知道自己要什么。

但他们不想学一整套工程栈，才能把一件小事做完。

过去两年，AI coding 工具变得很强。强到一个非工程师也会开始想：我是不是终于可以把软件工作外包给机器了？

但真正用起来，会碰到一个很奇怪的问题。

工具越强，越不敢放心让它跑。

因为它一旦开始动文件、跑命令、改配置、访问外部服务，你就会进入一个黑箱：它现在在干什么？它准备改哪里？这一步安全吗？它失败之后我还能回到哪里？它是不是拿着我的 token 在做我没看懂的事？

所以很多产品最后卡在两个极端。

一种很强，但太黑箱。  
它能做事，但你只能在旁边看一串日志，猜它是不是靠谱。

另一种很安全，但太弱。  
它像一个聊天框、表单工具、玩具 workflow，能帮你想，不能真的把软件工作做完。

Openwork 想解决的是中间这块。

不是“再做一个聊天框”。  
也不是“让 Agent 更自主”。  
而是把一次软件委托变成一个可控的工作单元。

你给它一个目标。  
它选择工作目录。  
它计划、读文件、改文件、跑命令、调用工具。  
危险动作停下来问你。  
做了什么留下记录。  
产物能看。  
失败了能知道失败在哪。  
之后还能回来继续。

这听起来不像一个酷炫功能。

但我越来越觉得，这才是 AI Agent 进入真实工作的缺口。

AI 不缺聪明。  
缺的是让人敢把工作交给它的工程边界。

我们内部把这层叫 harness。

如果不想听术语，可以把它理解成：给 Agent 装一个安全驾驶舱。

它不是把 AI 关起来，而是让 AI 做事时，有仪表盘、有刹车、有行车记录、有检查点。

一个真正可用的软件 Agent，至少要回答这些问题：

它要做什么？  
它已经做了什么？  
它改了什么？  
它现在卡在哪？  
这一步为什么需要我批准？  
如果它错了，我从哪里接回来？

这就是 Openwork 和很多 agent shell 的不同。

我们不把“模型会调用工具”当成产品终点。  
模型能调用工具，只是开始。

真正的产品在工具调用之后：

能不能把低层行为翻译成人看得懂的状态？  
能不能让每次运行都留下可检查的证据？  
能不能把读取、写入、外部调用分成不同权限？  
能不能让用户在关键时刻介入，而不是事后道歉？

这也是为什么 Openwork 不是要做一个泛泛的 extension marketplace。

我们当然有 extension runtime。  
现在已经接了 Apple Reminders、GitHub、Todo、Translate 这些 bundled extensions。

但更重要的不是“有多少扩展”。

更重要的是：这些能力给人用时，是一个清晰的界面；给 Agent 用时，是一个有权限边界的工具。

这里我们参考了两个方向，但没有照搬任何一个。

Raycast 做得很好的，是交互。

它让 AI 出现在你需要它的瞬间：快捷键、搜索框、Action Panel、Form、List、AI Chat。它像操作系统的一层命令面板。

但 Raycast 更像 invocation moment 的高手：你如何快速叫出能力、填参数、点确认。

长时间软件委托需要的东西更多。

它需要运行生命周期，需要 checkpoint，需要审批恢复，需要 artifact，需要 diff，需要工作目录和外部服务的边界。

Craft Agents 这类系统更强调 Source 和 Permission：某个外部能力可以读什么、写什么、在哪种模式下需要审批。

Openwork 现在更像这两者的组合：

Raycast 风格的 extension UI runtime。  
Craft 风格的 source/tool permission runtime。

翻译成人话就是：

给人操作时，它应该像一个自然的桌面工具。  
给 Agent 操作时，它必须变成一组受控能力。

read tool 可以默认可用。  
write / external tool 必须经过 permission 和 approval。  
账号、token、默认配置放在 Settings。  
模型不应该临时拿着 secret 自己发挥。

这件事听起来很工程，但它最后影响的是用户感觉。

用户不需要知道什么叫 permission runtime。  
他只需要感觉到：这个 Agent 不会突然越界。

用户不需要知道什么叫 artifact pipeline。  
他只需要在结束时看到：这些是它改的文件，这些是它产出的结果，这里是它等待我决定的地方。

用户不需要知道什么叫 checkpoint。  
他只需要在出错时不用从头再来。

这也是我们对 Openwork 的产品判断：

Raycast 赢在 invocation moment。  
Openwork 要赢在 delegated work lifetime。

从你发出任务，到任务执行，到中途审批，到产物出现，到失败恢复，到以后复查。

这一整段生命周期，才是我们想拥有的地方。

所以 Openwork 的第一版不会追求“Agent 团队”这种热闹叙事。

多个 Agent 当然可以有用。

但问题不是有几个 Agent 在干活。  
问题是一个用户目标能不能变成一个可控、可检查、可恢复的工作单元。

如果一个 Agent 能把事情做好，就一个。  
如果多个 Agent 真能降低风险、提高质量，再多个。

我们不想把复杂性摆在首页。

我们想把复杂性收进系统里，让用户看到一个简单循环：

说出目标。  
确认边界。  
看着它做。  
关键处批准。  
拿到结果。  
必要时恢复。

这是 Openwork 的产品核心。

一个安全、可控、面向非程序员的软件 Agent。

不是因为非程序员不聪明。

恰恰相反。

他们通常最懂问题本身，只是不想为了让软件听懂自己，而被迫变成软件工程师。

如果 AI 真的要改变软件生产，下一步不只是让程序员更快。

下一步是让更多懂问题的人，能安全地把软件工作委托出去。

这就是我们在做的东西。

Openwork。

让你把软件工作交给 AI，但不把方向盘交出去。
