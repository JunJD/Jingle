## Open Target Discovery Evidence

Codex 右上角的 `Cursor / Finder / Terminal` 不是 header 组件硬编码出来的三项，而是主进程 app-server 生成的 target read model。

关键链路：

- Renderer chrome：`thread-app-shell-chrome-BerBCXkf.js` 里用 `use-target-apps-D72Y95QQ.js` 拿 `preferredTarget`、`targets`、`availableTargets`，再用 `open-target-selection-DutvMofv.js` 选 primary target。点击 primary 或 dropdown item 时调用同一个 `open` 函数，传入 `target`、`appPath`、`openMode:"workspace"`、`persistPreferred`。
- Renderer query/action：`use-target-apps-D72Y95QQ.js` 查询 app-server method `open-in-targets`；点击后 mutate `open-file`。如果传了 URL path，会先用 `nativeBrowserDiscovery:"known"` 快速拿一版 native target，再刷新完整扫描结果。
- Main discovery：`main-CIL4OHS5.js` 的 `open-in-targets` handler 调 `OA(hostConfig)` 得到可用 target id，调 `yA()` 得到 target metadata，调 `MA(settingsStore, cwd, availableTargets)` 得到偏好 target，并返回 `{ preferredTarget, availableTargets, mode, targets }`。
- Target registry：同一个 main chunk 里 `uA=[...]` 是平台 target registry；`wA()` 遍历 registry 执行每个 target 的 `detect()`，`bA()` 返回检测成功的 target id，`yA()` 返回 label/icon/kind/hidden 等展示数据。
- Cursor：target 定义是 `id:"cursor"`、`label:"Cursor"`、`icon:"apps/cursor.png"`。macOS detect 会检查 `/Applications/Cursor.app`、`Cursor Preview.app`、`Cursor Nightly.app`，也会通过 `uO("Cursor")` 扫 `/Applications` 和 `~/Applications` 前缀匹配；最终还要求存在 `Contents/MacOS/Cursor` 和 `Contents/Resources/app/out/cli.js`，所以不是只看 app 名。
- Finder：target 定义是 `id:"fileManager"`、`label:"Finder"`、`kind:"fileManager"`，macOS detect 直接返回 `open`，args 走 folder/file path。历史偏好里旧值 `finder` 会映射回 `fileManager`。
- Terminal：target 定义是 `id:"terminal"`、`label:"Terminal"`、`kind:"terminal"`，macOS detect 通过 `/System/Applications/Utilities/Terminal.app` 是否存在决定是否可用，打开时使用 `open -a Terminal <dir>`。
- Native app：对 URL 或特定文件，main 还会走 `Gj()` / `Wj()`。`Gj()` 使用系统默认浏览器和 native browser discovery；`Wj()` 用 `native/launch-services-helper` 查某个文件可打开的应用。这类结果会映射成 `target:"systemDefault"`、`kind:"native"`、带 `appPath` 的 item。

映射到 Openwork：

- Header 组件不应该负责扫描 `/Applications`、调用 `open`、解析 app icon，甚至不应该知道 Cursor 是否存在。
- Openwork 本轮补了一个窄的 main/native open-target owner：`src/main/open-targets/service.ts` 负责 target discovery + open action；preload 暴露 `openTargets.list/openTargets.open`；renderer 的 `LauncherAiOpenTargetMenu` 只消费 target read model。
- 当前实现复用 launcher application search catalog 作为 app discovery 素材，但依赖方向停在 main/service；header 不扫描 `/Applications`，也不 import search provider 内部。

## Current Openwork Scope

本轮只允许实现：

- Launcher AI header chrome 的布局和局部菜单。
- Launcher AI sidebar shell 的本窗口 hover preview / click expanded。
- Open target 菜单识别并打开当前文件夹到 Cursor/Finder/Terminal 等可用目标。
- `pinned-ai-session` 独立 renderer/window kind，从当前 threadId 钉出一份会话窗口。
- Workspace/environment 当前真实事实展示。
- Thread menu 与 header chrome 的职责拆分。

本轮明确不实现：

- thread pin。
- sidebar preference store。
- sidebar thread projection。
- Dock menu。
- projector event 到 messages。
- opencode 协同基础设施。

如果某个按钮没有当前 Openwork owner，第一版必须 disabled 或 no-op；不能接旧 history page 或 history shell 的侧栏折叠状态，也不能为了纯 UI 临时新增 main/preload API。Open target 和 pinned session window 已经有独立 owner，不能回流到 workspace 或 header。
